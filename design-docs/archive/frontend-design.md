# 前端设计 — Graph Explorer Agent 交互界面

> 状态: 设计阶段 | 日期: 2026-06-03
>
> 本文档定义前端交互模型和信息架构。不涉及代码级实现细节（具体数据结构、组件库、路由方案）。

---

## 一、核心差异：不是聊天

通用聊天（ChatGPT/OpenClaw）是**对话驱动**——每次交互是一个 prompt。

Graph Explorer 是**任务驱动**——一次交互是多步工具调用 + 多轮推理 + 结构化复合产物。

关键 UX 后果：
- 输入需要参数化（不能纯靠自由文本猜）
- 30-120 秒的"静默期"里显示 Agent 的每步进展
- 输出是结构化产物（findings + threads），不是一段回复文本

---

## 二、交互模型：四个阶段

```
[参数确认] → [探索中] → [结果浏览] → [追问]
    ↑                                    │
    └────────────────────────────────────┘
          追问本质是新一轮探索，继承 State
```

### 阶段 1：参数确认

**原则**：聊天式输入 + Agent 主动确认。不是填表——是"你说，我确认"。
用户不需要理解 seed entity / max depth 这些概念。

```
Agent:  你想探索什么？

User:   芯片管制对英伟达供应链的影响

Agent:  了解。我整理了以下参数：
        · 探索目标：追踪美国对华芯片出口管制对英伟达供应链的传导影响
        · 起始实体：英伟达
        · 探索深度：3 跳（默认）
        需要补充或修改吗？

User:   加上台积电

Agent:  · 起始实体：英伟达、台积电 ✅

User:   开始吧

Agent → 进入探索中
```

参数确认不是一个 modal 弹窗——是对话的天然起点。用户多次来回修改参数直到满意。

### 阶段 2：探索中

**原则**：不 spinner。Agent 每步 tool call 结束后 push 进度，前端实时渲染。

```
[Step 1] 查询"英伟达"基本信息           [1.2s]
         → 发现 9 个事件，5 个聚类
         → 新实体: 台积电、微软、ARM、黄仁勋
         [展开▾]

[Step 2] 查询"台积电"基本信息           [0.8s]
         → 发现 6 个事件，3 个聚类
         → 新实体: 华为、中芯国际
         [展开▾]

[Step 3] 展开供应链相关聚类              [3.1s]
         → 6 个事件
         → 标识到 PC 战略加速信号
         [展开▾]

...

[Step 9] FINALIZE — 构建结果              [4.2s]

═══════ 探索完成 ═══════
```

每步一个进度气泡，默认折叠显示摘要，点击展开看完整细节（调了什么工具、返回了什么实体、产出了什么 intermediate finding）。

最后一步完成 → 进度气泡自然收拢 → 结果面板展开。

### 阶段 3：结果浏览

探索产物天然分两种：离散洞察（findings）和连续叙事（threads）。

**Findings 面板**：每条 finding 渲染为一张卡片。

卡片内容从上到下：
- 类型标记 + 信心标记（⚠ absence HIGH / ✓ concentration MEDIUM / 🔗 chain / ✗ pattern_violation）
- statement 正文（可读的一句话）
- 涉及实体列表（可点击 → 提示"追踪此实体"触发追问）
- evidence 条数（"5 条证据"）→ 点击展开 ku_id 列表 → 再点击 ku_id 弹出原文卡片
- conflict_with（如有）

排序：最直接回答 Goal 的排前面。

**Event Threads 面板**：每条 thread 渲染为一条时间线。

时间线内容：
- title + 一句话 summary
- 事件节点按时间排列：每个节点显示时间戳、实体、事件类型
- 节点之间标注关系类型和 reasoning（causal / temporal / entity_shared / contradiction）
- confidence 标记
- 时间线可缩放（整体 view）和展开（逐节点 detail view）

**布局决策**：Findings 面板和 Event Threads 并排，不上下堆叠。两种产物性质不同——离散 vs 连续——并排让用户同时看到两种视角。

**Stats 面板**（默认折叠）：步数、实体数、事件数、token 用量、完成原因、reliability note。

### 阶段 4：追问

**原则**：追问就是新一轮探索。同一个 Agent Loop，同一个 State。

与首次探索的唯一区别——初始 State 不为空：
- visited、findings、raw_event_archive 完整继承
- 新 goal = 用户的追问文字
- 新 seed 追加到已有 visited 集合中（已有的不重复查）

Agent 自己判断——数据够了就 sufficient 0 步回答；不够就调工具继续探索。不存在"分析模式"和"探索模式"的区分——这是同一个 loop 在数据充分/不充分时的自然行为。

**但有一层提示逻辑**：如果 Agent 判断需要深入探索（调多个 MCP 工具），前端在第一个 tool call 前给一个交互提示，告知用户将要发生等待。

```
User:   为什么没提三星？补查一下

Agent:  需要查询三星在芯片供应中的位置，预计 30-60 秒。
        [确认] [取消]

User:   确认

Agent → 进入探索中（进度气泡从 Step 10 开始）
```

追问结果**追加**到会话，不替换之前的探索产物。用户可滚动回看历史。多轮探索在同一会话中累积。

## 三、信息架构

一个会话画布的信息分层：

```
┌──────────────────────────────────────────┐
│ 会话标题栏                                │
│ Goal 简要 + 状态标记（探索中 / 已完成）    │
├──────────────────────────────────────────┤
│                                          │
│  历史气泡区                               │
│  · 参数确认对话                           │
│  · 探索 1: Step 1-9 进度气泡 (可折叠)      │
│  · 探索 1: 结果面板                        │
│  · 追问对话                               │
│  · 探索 2: Step 10-15 进度气泡 (可折叠)    │
│  · 探索 2: 结果面板                        │
│                                          │
├──────────────────────────────────────────┤
│  追问输入区                               │
│  [输入框: "在已有分析基础上继续追问..."]    │
│  [新会话] [导出报告]                       │
└──────────────────────────────────────────┘
```

**关键决策**：

1. **Evidence 不默认展开全文**。每条 finding 挂 2-5 条 ku_id，全文展开一屏装不下一张卡片。默认显示 evidence 条数，点击展开 ku_id 列表，再点击 ku_id 弹出原文弹窗。

2. **Stats 默认折叠**。探索结束后用户首先关心"发现了什么"，不是"花了多少步"。异常时才看 stats（步数异常多、token 超预算、降级完成）。

3. **多轮探索结果不替换，追加**。用户可能在滚动回顾第一轮 findings 的基础上提出第二轮追问。两轮结果都可见，形成分析脉络。

## 四、会话管理

**会话列表**（侧边栏或独立页面）：
- 每条会话显示：Goal 概要、起始实体、运行时间、findings 数量、threads 数量、最后活动时间
- 点击进入 → 恢复会话 → 可继续追问
- 支持删除、重命名

**会话持久化**：
- Agent 的 ExplorationState 随会话存储
- 追问时反序列化，Agent Loop 从已有 State 继续
- 历史会话不自动过期（分析性场景，用户可能几天后回来追问）

---

## 五、推送机制

**选推模式**。Agent Loop 每步 tool call 结束是一个天然的事件分界点——step 完成 → push 进度 → 前端更新气泡。

推送频率：每步 1 次，间隔 5-20 秒。远低于"实时"的粒度要求，不会造成压力。

每次 push 携带：
- 进度信息：第几步、decision、这一步做了什么
- 增量发现：这一步有没有 new finding、有没有新 entity
- 状态标记：运行中 / 已完成 / 出错

前端行为：
- 收到"运行中"→ 新增或更新当前 step 气泡
- 收到"已完成"→ 拉取完整产出 → 渲染结果面板
- 收到"出错"→ 显示错误信息 + 重试选项

## 六、边界情况

| 情况 | 交互 |
|------|------|
| 探索 0 finding（Agent 找不到任何洞察） | 显示空状态 + "该知识库范围内未发现明显模式" + 建议换角度或补充种子 |
| 探索过程中用户手动终止 | 保留已完成步骤的进度气泡 + 已有的 intermediate findings（如有） |
| 追问导致 Agent 陷入同一条路径 | Agent 自己检测 visited 重复 → 建议用户调整 goal 或方向 |
| 网络断开（推送中断） | 进度气泡停在最后收到的 step + "连接中断，重新连接中..." |
| FINALIZE 降级（LLM 失败） | 显示降级提示 + 代码直接输出的 findings + reliability note |
| Token 预算耗尽 | 显示"探索已终止（token 预算）" + 当前已发现的 findings + 提示缩小范围 |

## 七、与 OpenClaw 会话的关系

Graph Explorer 的前端是**独立界面**，不是 OpenClaw 的插件或嵌入。

原因：
- 交互模型不同：任务驱动 vs 对话驱动
- 渲染需求不同：结构化面板 vs 纯文本气泡
- 连接方式不同：WebSocket 步进推送 vs HTTP request/response

可以共存——OpenClaw 通过 MCP 调用 Graph Explorer（重型工具模式），用户在 OpenClaw 里通过 function call 触发探索。但直接使用 Agent 的前端是另一个独立入口。

## 八、待设计

### 8.1 LLM 驱动的参数确认（多轮对话）

当前实现：前端用简单正则从用户输入中提取 goal 和 seed entities，非 LLM 驱动。

设计目标：用户自然语言输入 → LLM 解析意图 → 提取结构化参数 → 多轮确认 → 启动探索。

需要的后端支持：
- 新增 `/api/sessions/:id/parse` 端点，接收自然语言，返回提取的 `{ goal, seed_entities, suggested_depth }`
- 或在 Agent Loop 之前加一个"参数确认 phase"，由 LLM 主导多轮对话

优先级：P2（当前简单解析可工作，LLM 驱动是体验升级）
